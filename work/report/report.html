Give a brief explanation of how the browser, the HTTP protocol and the web server interacts.

(a) the browser resolves the host name of the server from the URL; (b) The browser uses the domain name resolution mechanism to translate the host name of the server into the IP address of the server; (how to use IP instead of the domain name directly, steps a and b can be omitted) (c) the browser resolves the port number (if any) from the URL; (d) The browser establishes a TCP connection with the Web server through the IP address and port number; (e) The browser sends an HTTP request message to the server; (f) the server sends back an HTTP response message to the browser; (g) The connection is closed and the browser displays the document.

If you were to recommend a language/framework/technology to your potential employer and your next large project, what would it be and how would you “sell it”?

I would definitely recommend node.js for several reasons. 1. Concurrent connection, Asynchronous, event-driven model The asynchronous mechanism and event-driven process do not block the connection of new users, and do not need to maintain the connection of existing users. Based on this mechanism, in theory, there are user requests to connect in succession, and NodeJS can respond. Therefore, NodeJS can support higher concurrency than Java and PHP programs. Although maintaining an event queue also requires costs, because NodeJS is a single thread, events The longer the queue, the longer it takes to get a response, and the amount of concurrency will still be weak. NodeJS solves the problem of concurrent connections: change the way to connect to the server, each connection emits an event running in the NodeJS engine process and put it in the event queue, instead of generating a new OS for each connection Threads (and allocate some accompanying memory for them). 2. Non I/O blocking Java and PHP also have ways to implement parallel requests (child threads), but NodeJS does it naturally through callback functions (Callback) and asynchronous mechanisms.

Elaborate on a few (3-5) different security issues or concerns, from your own experience, related to web application security and relate them to the OWASP top ten.

I have really limited experience of dealing with security issues. So I do some research and conclude several most popular issues. 1.XSS (Cross-Site Scripting) is a cross-site scripting attack. Because abbreviations and CSS overlap, it can only be called XSS. A cross-site scripting attack is an attack performed by an illegal HTML tag or JavaScript running in the browser of a registered web site user with a security vulnerability. 2.SQL injection vulnerability (SQL Injection) is the most common type of security vulnerability in web development. You can use it to obtain sensitive information from the database, or use the characteristics of the database to perform a series of malicious operations such as adding users, exporting files, etc. It is even possible to obtain the highest permissions of the database and even system users. 3.Click hijacking is an attack method for visual deception. The attacker embeds the website that needs to be attacked by embedding it in their own webpage, sets the iframe to be transparent, and reveals a button in the page to induce the user to click. 4.The DDos attack is based on the DOS attack. It is easy to understand that dos is singled out and ddos is a group attack. Because of the development of modern technology, the lethality of the dos attack is reduced. So DDOS appeared. The attacker used a public network to combine a large number of computer equipment to attack one or more targets. From a technical perspective, DDoS attacks can be targeted at all layers of the network communication protocol. The methods are roughly: TCP-type SYN Flood, ACK Flood, Fraggle, Trinoo of UDP class, DNS Query Flood, ICMP Flood, Slowloris class and so on. These are secure issues I often heard about. I never suffer an attack before, but only by reading their principle and codes, I can imagine the harmful results and serious situations it may cause.

Summarize by providing your own guidelines to a new web programmer on the topic “The essential guidelines to follow to create secure web applications”.

1. Don't trust any external incoming data, perform related format checking, filtering and other operations on user input, and handle escape characters. The most common way is to escape the content of the input and output. Escape the parentheses, angle brackets, and slashes. 2. Use CSP content security policy. 3. X-FRAME_OPRIONS (X-Frame-Options) This can be said to be born to solve ClickJacking. It is an HTTP response header. It has a better support in modern browsers. It is to prevent clicks nested with iframe Hijack attack. The response header has three values, which are: (1) DENY indicates that the page is not allowed to be displayed by iframe, (the browser will reject any iframe loaded by the current page) (2) SAMEORIGIN indicates that the page can be displayed in the iframe under the same domain name, (the address of the frame page can only be a page under the same domain name) (3) ALLOW-FROM indicates that the page can be displayed in the iframe of the specified source 4. Block third-party websites from requesting interfaces, do not trust any incoming third-party data, use CORS, and set Access-Control-Allow-Origin.
